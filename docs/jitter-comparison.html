FREQ-TR JIT audit

Documento generado para comparar la implementacion actual de JIT en FREQ-TR contra la matematica de referencia en ECHO-TR y DISP-TR. Las graficas se calculan en el navegador con las constantes actuales del codigo.

Scope: diagnostico. No propone cambios automaticos ni modifica DSP. Sample rate usado en graficas: 48 kHz.

Resumen forense

Hallazgo principal: FREQ-TR comparte el mismo motor base de JIT que ECHO/DISP, pero ahora separa dos lecturas temporales: depthDelayMs = 1000 / FREQ para conservar profundidad y rateDelayMs comprimido para drift/flutter/tone. Esto evita que las frecuencias bajas hundan demasiado la velocidad del modulador.
La intensidad queda separada de la velocidad: FREQ-TR mantiene el suelo de profundidad sqrt(freq^2 + 60^2) y calcula esa profundidad desde el periodo real. La velocidad usa una compresion continua con minimo 4 ms, maximo 500 ms y exponente 0.80.
Diferencia conceptual: ECHO modula tiempo de lectura; DISP modula coeficientes/frecuencia equivalente; FREQ modula el incremento de fase de un oscilador. Aunque las constantes sean iguales, el resultado perceptual no puede ser identico sin una capa de remapeo especifica para FREQ.

Conclusiones operativas

Pregunta Respuesta desde los numeros
El motor de FREQ esta "mal portado" respecto a ECHO/DISP? No en constantes base: usa el mismo nucleo de metricas. Si hay diferencia, esta en la traduccion fisica: FREQ usa periodo de oscilador como tiempo equivalente.
Por que puede sentirse lento? Antes, a 1 Hz el motor veia 1000 ms. Ahora la profundidad sigue viendo 1000 ms, pero la velocidad usa un delay comprimido de ~190 ms.
Por que puede sentirse diferente a DISP? DISP calcula delay de grupo de una red all-pass. No equivale directamente a periodo de oscilador; stages y series pueden cambiar mucho la lectura temporal.
Que cambio seria estructural si se decide tocarlo? Ya aplicado: rateDelayMs comprimido para velocidad y depthDelayMs crudo para profundidad.

Formulas clave

FREQ-TR delay equivalente:
  depthDelayMs = max(0.05, 1000 / abs(freqHz))
  rateDelayMsRaw = clamp(depthDelayMs, 4, 500)
  rateDelayMs = 4 * pow(rateDelayMsRaw / 4, 0.80)

Metricas compartidas ECHO/DISP/FREQ:
  shortness = clamp(log2(500 / rateDelayMs) / log2(500 / 8))
  longness  = clamp(log2(rateDelayMs / 250) / log2(4000 / 250))
  flutterHz = clamp((4 + (130 - 4) * amount) * pow(250 / rateDelayMs, 0.90), 2, 7000)
  depthOct  = log2((depthDelaySeconds + depthSeconds) / depthDelaySeconds)

Aplicacion FREQ-TR:
  referenceHz = sqrt(abs(freq)^2 + 60^2)
  depthHz = referenceHz * (exp2(depthOct * 2.0) - 1)
  jitteredHz = sign(freq) * max(0, abs(freq) - out * depthHz)

Aplicacion DISP-TR:
  freqOctOffset = -out * depthOct * 2.0
  jitteredFreq = baseFreq * exp2(freqOctOffset)

Aplicacion ECHO-TR:
  targetOffsetSamples = baseDelay * (exp2(out * depthOct) - 1)
  readDelay = baseDelay + softSlew(targetOffsetSamples)
